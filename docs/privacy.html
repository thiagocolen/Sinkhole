Privacy & Data Security

At its core, Sinkhole is built on the principle of data ownership. However, syncing data to the cloud (Google Drive) or using third-party backup services (GitHub) introduces unique privacy challenges. This guide explores how to achieve true end-to-end encryption for your sensitive vaults.

The "Publicity" Problem

While Google Drive is secure, it is not zero-knowledge by default. If you use GitHub for secondary backups, making your repository public exposes your data to the world. Even private repositories are still hosted on third-party infrastructure.

Solution: Local Cryptography

The most effective way to secure your vault is to encrypt it before it reaches any semi-public platform. By using tools like File Encryptor, you can transform your readable files into tamper-proof binary blobs.

Recommended: Agentic Backup Workflow

You can automate your security using an autonomous agent (like Gemini CLI). Here is a professional workflow for maximum privacy:

  1. Sync to Drive: Use Sinkhole on Android to keep your local vault mirrored to Google Drive.
  2. Agentic Fetch: An agent on your PC pulls the latest changes from Google Drive.
  3. Recursive Encryption: The agent runs file-encryptor with a strong password to generate .enc files.
  4. Git Backup: The agent pushes the encrypted .enc files to a GitHub repository.

Implementation with File Encryptor

If you are using an agentic workflow, you can trigger encryption non-interactively:

# Set your secret password
export FILE_ENCRYPTOR_PASSWORD="your-strong-password"

# Encrypt your vault for GitHub backup
node file-encryptor/index.js encrypt ./my-vault --json

This ensures that even if your GitHub repository or Google Drive is compromised, your data remains an unreadable "sinkhole" to anyone without your master password.
